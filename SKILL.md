# 中文小说写作规范 (novel-writer)

本 skill 规范 AI agent 在编写、修改、扩展中文长篇小说时的工作流程。所有规则适用于以"卷-章节"结构组织的小说项目。

## 项目目录结构（强制）

每个小说项目的根目录必须遵循以下结构：

```
小说根目录/
├── 全局大纲.md            # 必需：整本小说的总大纲，按卷分割
├── 文风.md                # 必需：小说文风设定文件
├── 人物设定/              # 必需：角色设定文件夹
│   ├── 主角名.md
│   ├── 配角名.md
│   └── ...
├── 第1卷卷名/             # 各卷目录，命名为"第N卷卷名"
│   ├── 卷大纲.md          # 必需：本卷大纲，按章节分割
│   ├── 第1章章节标题.md
│   ├── 第2章章节标题.md
│   └── ...
├── 第2卷卷名/
│   ├── 卷大纲.md
│   └── ...
└── ...
```

开始任何写作动作前，必须先确认上述结构存在；若缺失，先与用户讨论并创建对应文件，再继续。

## 一、维护规范：全局大纲与人物设定

### 全局大纲（`全局大纲.md`）

- 必须存在于小说根目录
- 用简洁的语言描述整部小说的故事发展和主旨
- **必须按卷分割**，每卷一个章节，标题形如 `## 第N卷 卷名`
- 每卷描述至少包含：核心冲突、主要剧情走向、本卷主旨

### 人物设定（`人物设定/`）

- 必须以"人物设定"命名的文件夹
- 每个主要角色独立一个 `.md` 文件，文件名即角色名（如 `人物设定/林夏.md`）
- 每个角色文件**必须详细描述**：
  - **性格**：核心性格特点、行为偏好、说话方式
  - **背景**：出身、家庭、关键经历
  - **外貌**：体貌特征、穿着风格、辨识度细节
  - **在小说中经历的故事**：随剧情推进持续更新

### 一致性强制：剧情/设定冲突的处理流程

**触发条件**：用户对话中提到的剧情、角色行为或设定，与现有的全局大纲或人物设定相悖。

**处理流程（必须严格执行）**：

1. 检测到冲突时，**立即停止当前写作动作**
2. 用一段话明确指出冲突点：
   - 用户提到的内容
   - 现有大纲/设定中的描述
   - 具体的矛盾点在哪里
3. 询问用户：是否修改大纲/人物设定以适配新剧情？
4. 等待用户回复后处置：
   - 用户**同意修改** → 修改对应的大纲或人物设定文件，再继续写作
   - 用户**拒绝修改** → 调整新剧情以符合现有设定，或与用户进一步讨论
5. 修改完成后，确认本次写作未破坏其他文件的一致性

**严禁**：在未询问用户的情况下，自行选择"采纳新剧情"或"忽略新剧情"。

## 二、文风设置规范

### 文风文件（`文风.md`）

- 必须存在于小说根目录，文件名为 `文风.md`
- 在开始书写任何章节正文之前，**必须**先读取该文件
- 若 `文风.md` 不存在，**禁止**开始写正文；必须先向用户询问文风信息

### 向用户询问文风信息的流程

当 `文风.md` 不存在时，必须向用户提出以下问题（至少覆盖以下维度），得到足够明确的回答后才能创建文件：

1. **整体语言风格**：古雅/白话/通俗/网文风/严肃文学风？
2. **叙事视角**：第一人称/第三人称有限/第三人称全知？
3. **描写偏好**：偏重环境渲染还是偏重动作对白？心理描写的深度如何？
4. **节奏特征**：快节奏推进/慢节奏铺陈？章末是否留悬念？
5. **对话风格**：口语化/书面化？是否使用方言或特定时代用语？
6. **修辞习惯**：偏好何种修辞？排斥何种修辞？比喻的密度和类型？
7. **参考作品**：是否有希望对标或规避的同类作品？

**必须**将用户的回答整理为结构化的 `文风.md` 文件，至少包含以下章节：

- `## 整体风格`
- `## 叙事视角与口吻`
- `## 描写偏好`
- `## 节奏特征`
- `## 对话风格`
- `## 修辞与语言禁忌`
- `## 参考与规避`

文风文件创建后，后续写章节正文前**必须重读**，确保写作风格与之保持一致。

## 三、卷编写规范

### 新增卷的工作流（强制顺序）

**严禁**：在没有讨论卷大纲的前提下，直接创建卷目录或开始写章节。

**必须遵循的步骤**：

1. 与用户讨论本卷的核心冲突、主要剧情、章节划分（建议给出章节数量与每章主旨）
2. 用户确认大纲后，创建卷目录：`第N卷卷名/`（数字使用阿拉伯数字，如 `第3卷北境风云`）
3. 在卷目录下创建 `卷大纲.md`，写入与用户讨论一致的大纲内容
4. **同步更新** `全局大纲.md`，新增本卷概述部分
5. 完成上述四步后，才能开始写本卷任意章节正文

### 卷大纲（`卷大纲.md`）

- 必须存在于每个卷目录下
- **必须按章节分割**，每章一个二级标题，形如 `## 第N章 章节标题`
- 每章用简洁的语言描述：
  - **核心剧情**：本章发生的关键事件
  - **主旨**：本章承担的主题作用（推动主线、塑造人物、铺垫伏笔等）
  - **节奏**：开端、转折、结尾的安排

## 四、章节编写规范

### 章节正文与卷大纲的强制对应

**核心原则**：章节正文的剧情和节奏必须与卷大纲中对应章节的描述**完全保持一致**。

执行流程：

1. **写作前**：
   - 读取 `卷大纲.md` 中对应章节的描述
   - **读取 `文风.md`，确认文风要求**
2. **写作中**：严格按照卷大纲的剧情走向、节奏推进；细节、对话、描写可自由发挥，但骨架不可偏离；**语言风格必须符合文风.md的设定**
3. **写作后**：核对正文与卷大纲是否对应；不对应则按"反向同步规则"处理

### 用户修改正文的反向同步规则

**触发条件**：用户对已写好的章节正文做出实质性修改（剧情走向、关键事件、人物行为、节奏等）。

**必须执行的反向同步**：

1. 识别用户修改的具体内容
2. **立即修改 `卷大纲.md`** 中对应章节的描述，使其与新正文一一对应
3. 如修改影响后续章节的剧情，主动询问用户是否需调整后续章节的卷大纲
4. 如修改影响人物（性格、关系、经历），同步修改 `人物设定/` 中的对应文件
5. 如修改影响整卷或全书走向，同步修改 `全局大纲.md`

**严禁**：保留与正文不一致的旧大纲，让大纲与正文脱节。

### 字数检查（每章必做）

**触发条件**：完成一章正文的编写或大幅度修改后。

**必须执行的步骤**：

1. 使用 `wc -m` 统计字符数：

   ```bash
   wc -m "第1卷某卷名/第3章章节标题.md"
   ```

2. 检查字符数是否在 **3000-5000** 之间
3. 处理结果：
   - **少于 3000**：扩写。补充细节描写、对话、心理活动、环境描写等，但不得偏离卷大纲
   - **多于 5000**：精简或拆分。先尝试删减冗余描写；若内容确实承载过多剧情，与用户讨论是否拆为两章（拆章必须同步更新 `卷大纲.md`）
   - **3000-5000 之间**：通过

说明：`wc -m` 统计字符数（含标点、换行）。中文小说习惯将其视为"字数"。如需排除空白字符以更精确，可用：

```bash
tr -d '[:space:]' < "第1卷某卷名/第3章章节标题.md" | wc -m
```

## 五、章节 Review 规范（强制）

### 触发条件

**每当完成一个章节的正文写入时，必须启动一个上下文隔离的子 agent 进行 review。不得跳过。**

### Review 子 agent 的启动方式

使用 Agent 工具启动子 agent，**必须**设置 `isolation: "worktree"` 以确保上下文隔离。子 agent 的 prompt 必须包含以下全部指令。

### Review 检查项（必须全部执行）

子 agent 收到章节文件路径后，按以下顺序执行检查：

#### 1. 字数检查

- 使用 `wc -m` 统计该章字符数
- 结果必须在 **3000-5000** 之间
- 若不在范围内，标记为不通过，并给出具体字数

#### 2. 内容检查

- 读取该卷 `卷大纲.md` 中对应章节的描述
- 判断章节正文内容是否与卷大纲描述**相符**
- **严禁**正文延伸到后续章节的剧情
- **严禁**正文包含尚未在卷大纲中规划的内容
- 若发现内容越界或偏离，标记为不通过，指出具体偏离点

#### 3. 文风检查

- 读取根目录 `文风.md`
- 判断该章节正文的语言风格、叙事口吻、描写方式、对话风格等是否符合 `文风.md` 的规范
- 若存在明显偏差，标记为不通过，指出偏差之处

#### 4. AI 风格检查

**此检查必须严格执行，杜绝一切 AI 风格。**

检查前，可先用 humanize 技能进行初步检查。之后**必须**手动逐项检查以下 AI 风格指标：

- **比喻密度**：
  - 同一句中不允许出现 **超过 2 处** 比喻
  - 同一章节中不允许出现 **超过 3 处** 比喻
- **排比密度**：
  - 同一章节中不允许出现 **超过 3 处** 排比
- **"不是，是"句式**：
  - **严禁**出现 "不是……是……" 这类 AI 典型句式
- **缩写与词语简化**：
  - **严禁**出现意义不明的缩写（如用莫名缩写代替正常词汇）
  - **严禁**对词语进行不必要的简化（如"方"代替"方向"、"情"代替"情况"等）
- **语言规范性**：
  - 必须尽可能使用**通俗易懂**的词汇
  - 必须完全符合**普通话语言习惯**，禁止生造词、生硬翻译腔或不符合中文表达习惯的措辞
- **短句堆砌**：
  - 场景描写和人物动作描写**禁止堆砌短句**
  - 应当充分展开描绘，使用完整、流畅的句子呈现画面和动作，而非用碎片化短句罗列
- **"还在"句式滥用**：
  - **严禁**大量使用 "还在" 句式进行背景或状态描写
  - **禁止**出现大量类似 "血还在流，她还在看" 这种空洞、重复、无意义的连续描写
  - 对背景状态应进行正常、具体的描绘，而非用 "还在" 敷衍带过

若发现上述任何一项违规，标记为不通过，逐条列出违规位置和原文。

### Review 结果处理

子 agent 完成全部检查后，返回一份 Review 报告，格式如下：

```
## Review 报告：第X卷/第Y章 章节标题

### 字数检查：[通过/不通过]
- 实际字数：XXXX

### 内容检查：[通过/不通过]
- （若未通过，列出偏离点）

### 文风检查：[通过/不通过]
- （若未通过，列出偏差之处）

### AI 风格检查：[通过/不通过]
- （若未通过，逐条列出违规项及原文）

### 总评：[通过/需修改]
```

**主 agent 收到 Review 报告后的处理**：

- **总评为"通过"**：确认该章节完成，可继续后续工作
- **总评为"需修改"**：根据 Review 报告中指出的问题修改章节正文，修改完成后**必须重新启动 Review 子 agent 再次检查**，直至通过

## 六、写作前/写作后检查清单

每次开始写作或修改章节时，按顺序检查：

```
写作前：
- [ ] 已读 全局大纲.md，理解全书走向
- [ ] 已读 本卷 卷大纲.md，理解本卷剧情
- [ ] 已读 涉及角色的 人物设定/角色名.md
- [ ] 已读 文风.md，确认文风要求
- [ ] 用户提到的内容与上述文件无冲突；如有冲突，已询问用户

写作中：
- [ ] 严格遵循卷大纲对该章的剧情和节奏描述
- [ ] 语言风格符合文风.md的设定

写作后：
- [ ] 已用 wc -m 统计字数，结果在 3000-5000 之间
- [ ] 已启动上下文隔离的 Review 子 agent 进行检查
- [ ] Review 报告总评为"通过"
- [ ] 用户修改正文后，已同步修改 卷大纲.md（必要时连带人物设定、全局大纲）
- [ ] 全局大纲、卷大纲、人物设定、文风、章节正文五者无矛盾
```

## 七、关键禁止事项

- **禁止**在没有 `全局大纲.md` 的情况下开始写小说；应先与用户讨论并创建
- **禁止**在没有 `文风.md` 的情况下写章节；若缺失必须先询问用户并创建
- **禁止**在没有 `卷大纲.md` 的情况下写章节；应先与用户讨论并创建
- **禁止**自行决定与已有大纲/设定/文风相悖的新剧情或新风格；必须询问用户
- **禁止**保留大纲与正文不一致的状态（任一文件被修改后，必须双向同步）
- **禁止**跳过 `wc -m` 字数检查
- **禁止**跳过章节 Review（上下文隔离子 agent 检查）
- **禁止**在新增卷时跳过"先讨论卷大纲再建目录"的顺序
